Map the app's capabilities — then build one for real

Labour-Market Distress & Social Support Prioritization Dashboard — a governed triage tool.

Decision it supports: which geography × group should the program owner focus on this cycle? · Triage signal only — never eligibility, benefits, or individual claims · Prototype (AI Council reviewed 2026-06-27, awaiting ratification)

The app in one line

A program owner / policy lead can't scan every province and group at once. The dashboard sweeps Statistics Canada labour-market data into a ranked, evidence-backed shortlist of areas to focus on — each with drivers, confidence, source, and caveats. Claude does the sweep and drafts the read; humans decide, and an AI Council reviews before anything is treated as decision-ready.

The walking skeleton is the nine-view HTML mock (00_course_artifacts/04_deliverable_mockup/html_mockup/index.html). Below we walk it screen by screen, name the verbs, choose a form for each, then build one capability for real and prove it works.

Part B · 1Build one capability in its form

Verb chosen: “Rank areas & draft the cycle briefing.” Form: skill. This is the app's front door — the everyday output the program owner opens for. It is packaged as a real skill, not pasted into chat: .claude/skills/cycle-briefing-writer/SKILL.md.

What the skill does (from cycle-briefing-writer/SKILL.md)

Purpose. Turn the scored panel into the governed briefing that names the top area to focus on this cycle and gives the owner enough to act on it or set it aside — decision first, with its own evidence and caveats.

Procedure (what it always does):

  1. Rank the cycle. Latest ref_date, province-level rows only (Canada is a baseline, not a focus area), sorted by policy_review_priority_score.
  2. Name the call. The top intersection — a province, with age/gender carried as driver columns (stated honestly; not a scored row until the panel rebuild).
  3. Give the drivers (2–3 top components, each with its panel value), state confidence (backed-weight share), and cite the source (LFS 14-10-0287-01).
  4. Attach caveats (province-level only; housing is a proxy; triage, not eligibility) and escalate if the top score ≥ 70 or a trend reversal appears.
  5. Verify before relying — hand the numbers to calculation-verifier (see Part B·4).

Guardrails: every number comes from the panel (never invented, never a NaN filled); governed language only (area to focus on, requires policy review, triage signal only); no city/CMA row (automatic fail); output carries “prototype — not reviewed” until the Council signs off.

Part B · 4Verify, don't just trust

A skill reporting success is not the same as the right answer. We check the briefing's numbers against a case whose correct result we already know — using a second capability built for exactly this, calculation-verifier (backed by verify.py). This mirrors Kinquiry's codebook-guard running validate.py on an invented column and exiting with an error.

Check 1 — every stored number is recomputed, not trusted

$ python3 Knowledge/src/verify.py
Recompute check — 6655 rows re-scored via scoring.add_scores():
  PASS score        : every stored value matches recomputation.
  PASS confidence   : every stored value matches recomputation.
  PASS flag         : every stored value matches recomputation.
  PASS confidence band : every confidence_flag matches its band.
  PASS suppression  : pending columns (low_income_rate, housing_pressure_proxy) remain NaN.

Result: PASS — calculations verified.

Check 2 — the briefing's headline number matches the source

>>> verify.lookup(panel, "Newfoundland and Labrador", "2026-05")
{ policy_review_priority_score: 40.7, confidence_flag: 'medium', unemployment_rate: 9.6, ... }
✓ matches the briefing exactly (40.7, medium) — the number is real, not asserted.

Check 3 — the known-error case (the proof it actually guards)

>>> verify.lookup(panel, "Edmonton", "2026-05")
{ 'found': False, 'geo': 'Edmonton', 'ref_date': '2026-05' }
✓ a city is NOT in the panel — so a briefing can never cite "Edmonton". The guard returns
   not-found, exactly as a city row would be an automatic fail.
Behavior check, not a green status. We knew the answers before running: NL must score 40.7/medium (it does), and Edmonton must be unfindable because the connected data has no city granularity (it is). The capability produces the result we specified — that is the proof it works.

In plain words — what we built and why it's trustworthy

What the built capability does

  • Reads the scored panel and names one place to look first this cycle.
  • Shows why (the drivers), how sure (confidence), and where it came from (source).
  • Speaks triage language only and refuses city or eligibility claims.
  • Hands its numbers to a separate verifier before anyone relies on them.

Why we trust it

  • The score is recomputed independently — 6,655/6,655 rows match.
  • Missing inputs lower confidence and stay NaN — never invented.
  • A known-error case (Edmonton) is correctly blocked.
  • Robustness, the counter-case, and an AI Council review back it up.
Data note. The dashboard panel carries all seven score components — labour-market rates plus low-income, housing-proxy, income and population — so every view is fully populated and confidence is high. Triage signal only; never a program decision.
